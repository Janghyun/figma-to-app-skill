# figma-to-app — Figma 링크 1개 → 동일 품질 웹앱

## Overview

Figma 링크 하나를 받아 화면 열거 → 계약 고정 → 병렬 포팅 → QA 루프로 일관 품질의 웹앱을 만든다.
**핵심 원칙: 병렬화 전에 계약(디자인 토큰·셸·컴포넌트 API)을 먼저 고정한다.** 계약 없이 병렬화하면 화면마다 색·셸이 미묘하게 달라진 앱이 나온다.

역할 분담: 이 스킬은 **절차**만 담는다. 프로젝트 사실(토큰 값, 임시 브랜드명, 화면 카탈로그 등)은 대상 레포의 `CLAUDE.md`·`docs/`에 기록해 미래 에이전트가 자동으로 읽게 한다.

## 입력 파싱

`https://www.figma.com/design/<fileKey>/<파일명>?node-id=<A-B>&m=dev`
→ fileKey, 시작 노드 `A:B` (URL의 `-`는 `:`로). `m=dev`는 뷰 표시일 뿐 MCP 접근과 무관.

## 0단계. 접근 확인 — 여기서 절반이 결정됨

1. Figma MCP 도구 존재 확인 (ToolSearch `+figma`). 없거나 끊겼으면: `authenticate` 도구가 있으면 OAuth URL 생성 → 브라우저 자동화로 승인 (사용자가 Figma에 로그인돼 있으면 에이전트가 직접 완료 가능). 도구 자체가 없으면 사용자에게 Figma MCP 연결을 요청하고 멈춘다.
2. `get_metadata(fileKey)`로 페이지 목록 시도 → 성공 여부가 1단계 경로를 결정.

## 1단계. 정찰: 링크 1개 → 화면 전체 목록 (3중 fallback)

1. 페이지 목록 API 정상 → 페이지별 순회하며 화면 프레임 열거
2. 고장(페이지 일부만 반환 포함) → `use_figma`(figma-use 스킬 먼저 로드)로 `return figma.root.children.map(p=>({id:p.id,name:p.name}))` 실행 → 전체 페이지 열거 (검증됨: REST 열거가 1페이지만 반환한 파일에서 19페이지 전부 확보). 이후 페이지별 `get_metadata(pageId)`는 정상 동작
3. 그래도 부족 → 받은 노드에 `get_metadata`로 캔버스 역추적, 최후엔 사용자에게 "Pages 패널 우클릭 → Copy link" 요청

**카탈로그 서브에이전트**: 화면 프레임 vs 문서 크롬(플립북 헤더·Description 패널·플로우차트·메모) 분류, 스크린샷을 `docs/figma/<노드ID>.png`로 저장, `노드ID | 화면명 | W×H | 섹션 | 한줄설명` 표 산출. **이 표가 이후 모든 단계의 작업 지시서다** — 섹션 열은 4단계에서 IA가 없을 때 네비·라우트 구조가 된다.

드래프트 보드(정리 안 된 파일) 신호: 한 캔버스에 폭이 다른 세대 혼재(예: 1440 와이어프레임 + 1920 상세), 동명 상태 변형 다수, 흩어진 메모 텍스트. → 좌표 클러스터링 + 인접 텍스트 라벨로 화면↔섹션 매핑.

**사람 게이트 (필수, 건너뛰지 말 것)**: 세대가 혼재하면 어느 세대가 정본인지, 상태 변형(로딩/실패/빈 상태 등)을 전부 만들지 대표만 만들지 — 두 질문을 묶어 한 번에 사용자 확정을 받고 진행한다. 사용자가 답 없이 "그냥 진행"만 원하면 기본 가정(최대 폭 세대 = 정본, 대표 상태만)을 명시하고 진행. 탈락 세대·미구현 변형은 구현에서 제외하되 스펙 소재(SpecMarker 비고)로만 참조. 뷰포트는 정본 폭 기준 데스크톱 고정(반응형은 별도 요청 시).

**디자인 공백(gap) 게이트**: 카탈로그·Phase A에서 "필요하지만 Figma에 UI가 정의되지 않은 것"을 발견하면 — 정본 화면 없는 라우트(와이어프레임/IA/플로우 다이어그램만 존재), 도달성에 필요한 미정의 인터랙션(예: 아이콘 레일 사이드바의 하위 메뉴 펼침), 필수 요소 부재(로그인 등) — **임의로 고안해 넘어가지 말고** 공백 목록을 모아 사용자에게 항목별 옵션을 제시해 선택받는다: (a) 기존 화면 패턴 차용 재해석 구현 (b) 최소 구현(placeholder/툴팁 수준) (c) 미구현 + 기록만. 무응답·"그냥 진행" 시 기본값 = (a) 재해석. 선택 결과는 CLAUDE.md의 **"디자인 공백 결정"** 섹션에 기록해 병렬 에이전트 전원이 따르게 하고, 어떤 선택이든 해당 UI에 SpecMarker로 "디자인 미정의 — 재해석/최소구현, 디자이너 확정 필요"를 명시한다.

## 2단계. 기반 계약 — 재사용 판단 먼저

- 기존 앱(토큰·셸 보유)이 있으면: 대표 화면 1개 `get_design_context`로 색·폰트를 기존 토큰과 대조 → 같은 디자인 시스템이면 기존 토큰·AppShell·SpecMarker 재사용, 화면만 추가 (라우트 추가 or 별도 브랜치)
- 다르거나 신규면: **Phase A** — 에이전트 1명이 (빈 레포면 스캐폴딩부터: 기본 Vite+React+TS+Tailwind) 대표 화면 1개를 분석해 토큰(예: surface/ink/muted/brand…)·공통 셸(사이드바+헤더 = AppShell 단일 소유)·SpecMarker 등 공통 컴포넌트 API·폰트를 확정하고 대표 화면 1개를 완성해 계약을 증명한다. 대표 화면 = 셸·토큰 커버리지가 가장 넓은 화면(보통 대시보드류).
- 확정된 계약은 대상 레포 `CLAUDE.md`에 기록.

## 3단계. 병렬 포팅

- **화면 1개 = 에이전트 1개**, 파일 소유권 명시("네 파일만 수정")
- 전원 동일 프롬프트 템플릿:

```text
[화면명]을 src/pages/<File>.tsx 로 포팅하라. 네 파일만 수정한다.
- Figma 노드: <fileKey> / 대표 <노드ID> (+상태 변형: <노드ID 목록>) — get_design_context + get_screenshot 사용
- 참조 스크린샷: docs/figma/<노드ID>.png
- 상태 변형을 구현할 경우 한 페이지의 상태로 묶는다 (로컬 상태 토글 또는 ?state= 쿼리)
- 색·폰트는 이 토큰만 사용: <토큰 목록>. 임의 hex/기본 팔레트 금지
- 셸(사이드바·헤더)은 AppShell이 소유 — 화면은 내용물만 만든다
- 문서 크롬(플립북 헤더·Description·메모)은 제거, 스펙은 <SpecMarker n= title= body=/>로
- import는 상대 경로. 완료 기준: tsc 통과. 보고는 요약만
```

## 4단계. 통합

전체 `tsc` + `build` → 라우팅 + 2depth 네비 연결(IA가 있으면 그것을, 없으면 1단계 카탈로그의 섹션 매핑을 구조로 사용) → 주요 CTA를 라우트에 wiring.

## 5단계. QA 루프 — 최소 2바퀴

헤드리스 브라우저(Playwright 우선, 없으면 브라우저 MCP)로 **전 라우트**: ① 렌더·콘솔 에러 ② 클릭 도달성(요소가 실제로 클릭되는가) ③ 중복 크롬(화면이 자체 헤더/사이드바를 또 만들지 않았는가) → 수정 → **재QA**. Description/스펙이 있으면 마커 오버레이 표시도 확인. 최소 2바퀴이며 무결이 될 때까지 반복 — 수정이 새 문제를 만든다.

## Quick Reference

| 단계 | 산출물 | 병렬성 |
|---|---|---|
| 0 접근 확인 | MCP 연결 + 페이지 목록 가능 여부 | - |
| 1 정찰·카탈로그 | 노드ID·화면명·W×H 표 + 스크린샷 (+사람 게이트) | 서브에이전트 1 |
| 2 기반 계약 | 토큰·AppShell·컴포넌트 API + 대표 화면 1개, 레포 CLAUDE.md | 에이전트 1 |
| 3 병렬 포팅 | 화면별 페이지 파일 (각자 tsc 통과) | 화면 수만큼 |
| 4 통합 | 라우팅·네비·CTA wiring, build 통과 | 직렬 |
| 5 QA 루프 | 전 라우트 무결 (≥2바퀴) | 직렬 반복 |

## Common Mistakes

| 실수 | 결과/교정 |
|---|---|
| 계약 없이 바로 병렬화 | 화면마다 파랑이 다른 앱 (#4337ff vs #4436fe vs blue-600) → Phase A 먼저 |
| 화면이 자체 헤더·사이드바 생성 | 셸 중복 → 템플릿에 "AppShell 소유" 명시, QA ③에서 검출 |
| 카탈로그 없이 포팅 시작 | 화면 누락·문서 크롬 혼입 → 1단계 표가 작업 지시서 |
| 사람 게이트 생략(세대·상태 변형 임의 선택) | 잘못된 정본으로 수십 화면 낭비 → 한 줄 확정 후 진행 |
| QA 1바퀴로 종료 | 수정이 만든 회귀 미검출 → 최소 2바퀴 |
| 상태 변형 프레임을 별개 화면으로 오인 | 화면 수 폭증 → 동명 변형은 한 화면의 상태로 묶기 |
